Design Patterns

What is a pattern?

A design pattern is a general reusable solution to common problems in software design. It is a template to solve a problem that can be used in many different situations. The patterns concept originated in architecture in 1977, and was introduced as a programming concept in 1994 in the book Design Patterns: Elements of Reusable Object-Oriented Software.
There are three types of patterns:

  • Creational
  • Structural
  • Behavioral

Pros / Cons

Design patterns gives the developer a tested and proven strategy for solving software problems. This can speed up development as well as mitigate future problems as the software grows larger. However design patterns can overcomplicate certain designs when used incorrectly.

Factory Method

Also known as: Virtual Constructor

Source

Intent

Factory Method provides an interface for creating objects in the parent class, but allows subclasses to alter the type of the objects that will be created.

Problem

You are creating a logistics management app. First version of your app will handle only a single object, but in the future we will want to expand the functionality and alterations.

After a while your code will be coupled with the original object and making changes will be hard.

Solution

Factory Method replaces direct object creation with a special "factory" method. The constructor should be moved inside that method. Objects returned by the factory method are often called "products".

May seem pointless at first, but now we can override the factory method in a subclass and change the class of the object that will be created.

There is slight limitations: all products must have a common interface. Factory method is a base class that should be returning this common interface.

E.g. It is like having a module object parent class and then making slight alterations in the children that inherit from this parent class.

Application

References

This allows user to extend existing code as oppose to modifying. This also condenses code to produce components to a single creational method. We can save system resources and reuse existing objects, instead of rebuilding new ones.

Pros & Cons

Pros Cons
Follows the Open/Closed Principle Requires extra subclasses.
Avoids tight coupling between concrete products and code that uses them.
Simplifies code due to moving all creational code to one place.
Simplifies adding new products to the program.

Relations with Other Patterns

Most designs will start with Factory Method, but slowly evolve towards Abstract Factory, Prototype, or Builder as designers need more flexibility.

Abstract Factory classes are often implemented with Factory Methods, but also used in Prototype.

Factory Method can be used along with Iterator pattern to return proper iterators based on subclass.

Prototype requires initialize, but not subclassing and Factory Method is the opposite

Factory Method focuses on Template Method and is often served as a step in a large Template Method.
